If you keep all of this in the same layer, the app will become very hard to maintain.

What I recommend structurally
1) Make mode a first-class concept
Every attempt/session should know:

mode: 'exam' | 'cpd'
status: in_progress | paused | completed | ended
checkpointIndex
heartsUsed
adsWatched
xpEarned
gemsEarned
That means the quiz session is not just “a quiz”; it is a stateful learning run.

2) Add a dedicated session/checkpoint system
This is critical for Exam mode.

You need to persist:

current question index
selected answers
score so far
remaining questions
whether results are hidden until end
For Exam mode
save checkpoint every X questions or after each question
allow resume from last checkpoint
no final results until quiz completion
if user exits, continue from saved state
if user ends quiz manually, delete from DB
For CPD mode
only 10 questions per quiz
show per-question feedback immediately
save progress continuously
ad at the end unless hearts run out sooner
This should live in a separate session service, not inside the UI.

3) Introduce an economy service
Hearts, gems, boosts, and premium rules should be centralized.

Suggested rules:

hearts consumed for attempts in CPD
battery/hearts maybe shared across modes if you want one currency
ads can refill hearts
premium bypasses ads and disables heart limits
gems earned through referrals, quests, streaks, rewards
gems can be exchanged for hearts
gems can be gifted to friends
XP boosts apply during timed windows
This needs a dedicated backend service because it will affect:

quiz access
ad rewards
referral rewards
premium entitlement
premium purchase flow

4) Split quiz UX by mode
Exam mode UI
more serious
checkpoint indicator
ad breaks after every 10 questions or hearts exhausted
hidden results until end
resume button
progress bar with total questions remaining
CPD mode UI
more interactive
question-by-question feedback card/modal
reward animations
10-question sessions
ad at end or when hearts are exhausted
daily goals / streak / XP / badges visible

What to add for sounds
You should treat sound as another shared system, not part of quiz scoring.

Sound types to support
correct answer sound
wrong answer sound
streak achieved sound
badge unlocked sound
level up sound
heart lost sound
heart refilled sound
ad reward sound
quiz complete sound
checkpoint saved sound
mode-specific ambience or cue sounds
Recommended structure update
Add a new domain:


lib/audio/
And maybe:


components/audio/
Example files
lib/audio/types.ts
lib/audio/sounds.ts
lib/audio/player.ts
lib/audio/sound-events.ts
Why this is important
If you hardcode audio into quiz components, it will get messy fast.

Instead:

quiz or gamification logic emits an event
audio layer listens and plays the correct sound
UI stays clean
Suggested approach
Use a small sound event map like:

answer_correct
answer_wrong
streak_up
badge_unlock
level_up
heart_loss
heart_refill
reward_claim
quiz_complete
checkpoint_saved
Then in both exam and cpd modes, the app can trigger different sounds depending on the event.
